カプセル化とは


アクセス修飾子には「public」「private」「protected」の3つがありますが、不用意にフィールドやメソッドに対して「public」を付けると、他の全てのクラスやメソッドから好き勝手にアクセス出来るようになるため誤動作やバグの原因になります。
しかもデバッガを使わないとどこからアクセスされたのかすぐには分からないのでデバッグ作業も困難になります。

従って一般的には「public」の使用は最低限に留め、原則的にフィールドやメソッドのアクセス権は「private」にします。

ただし全てを「private」にするとサブクラスを作った時にフィールドやメソッドにアクセス出来なくて不便な時がありますので、その場合は「protected」にします(※)。

※ ただし「protected」はサブクラスから見たら「public」と同じなので使ってはいけないと主張する人もいます。

この様にフィールドやメソッドのアクセス権を原則「private」や「protected」にして他のクラスから簡単にアクセス出来ないようし、そのクラスの中で何をしてるのか隠蔽して見えなくすることを「カプセル化」といいます。

カプセル化されたクラスは「public」を付けたメソッド経由でしか他所からはアクセス出来なくなるので、独立性が上がってバグの影響をそのクラスの中だけに留めておけるようになるためデバッグ作業が楽になります。
同様に、あるクラスに行なった変更が他のクラスに及ぼす影響も小さくなるので、多人数での開発がやりやすくなります。

一般社会では「隠蔽」は悪い意味で使われていますが、オプジェクト指向の世界では「隠蔽」は良いことですので、クラスは積極的に隠蔽して行きましょう。
