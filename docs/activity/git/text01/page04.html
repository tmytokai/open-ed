アクティビティ: git によるバージョン管理
学習項目: [1] git とは

ワーキングツリー/HEAD/インデックス

git を利用するためには最低限「ワーキングツリー」「HEAD」「インデックス」という３つの用語を知っておく必要があります。


1. ワーキングツリー

まず「ワーキングツリー」とは、実際にユーザーが作業しているフォルダのことでワークツリーとも言います。

ワーキングツリー内にあるファイルは git に管理させることができるので(※)、ファイルを更新したらローカルリポジトリにコミットして保存することが出来ます。

※ ワーキングツリー内のどのファイルを git で管理させるかはユーザーが手動で指定します。


2. HEAD

次に「HEAD」とは、ローカルリポジトリの最新コミットを示す識別子のことです。

コミットを実行すると、git は管理化にあるファイルのその時点におけるスナップショットを作成してローカルリポジトリに保存します。
このスナップショットの事を正確には「コミットのスナップショット」というのですが、一般的には略して単に「コミット」とだけ呼んでいます。

それでコミットを繰り返していくとコミット(のスナップショット)が複数出来るのですが、最新のコミット(のスナップショット)のことを「HEAD」といいます。


3. インデックス

最後に「インデックス」とは、コミットする前に更新したファイルを一時保存しておく場所のことでステージとも言います。

インデックスにファイルを一時保存しておけば、いつでも保存した時の内容にファイルを復元できます。

もちろんインデックスを使わずいきなりコミットしても良いのですが、ちょっとだけ更新したファイルを一時保存したいと思った時、いちいちコミットしていたらスナップショットが増えすぎて後で訳が分からなくなります。
そこで通常はインデックスにファイルを一時保存しておいて、ある程度作業が進み、まあこんな物で良いかと思ったらファイルをコミットします。

なお、上で「コミットするとスナップショットを作成してローカルリポジトリに保存」と書きましたが、正確にはインデックスの内容でスナップショットを作ります。

またインデックスにファイルを一時保存することを「インデックスに登録する」とか「add する」とか「ステージングする」とか言います。

以上の「ワーキングツリー」「HEAD」「インデックス」は言葉で説明すると非常に分かりにくい概念なのですが、次のテキスト以降でビジュアル的に説明しますので、良く分からなかった人はひとまず次のテキストに進んで下さい。
