git を使うと「ローカルリポジトリ」と「リモートリポジトリ」をうまく使い分けて、次のような流れで効率的にチーム開発を進めることができます。

(ステップ1) ある開発者は、自分に割り当てられた作業中の適当なタイミングでローカルリポジトリに対して更新したファイルをコミット(※)する。

※ commit・・・更新したファイルの内容をローカルリポジトリに保存すること。一度コミットすればいつでもコミットした時点の状態にファイルを戻す事ができる。

(ステップ2) その開発者は、割り当たった作業が全て完了するまでステップ1を繰り返す。

(ステップ3) 自分の作業が終わったら、その開発者はリモートリポジトリに対してプル(※)を実行する。

※ pull・・・他の開発者がリモートリポジトリにアップした更新内容をダウンロードして取り込むこと。

(ステップ4) 衝突(※)が起きたら修正する。

※ conflict・・・他の開発者の更新内容と自分の更新内容が被ること。

(ステップ5) 衝突が無くなるまでステップ3と4を繰り返す。

(ステップ6) リモートリポジトリに対してプッシュ(※)を実行する。

※ 自分の更新内容をリモートリポジトリにアップすること

(ステップ7) ステップ1に戻り新しい作業を開始する。

この図の様にリモートリポジトリがクラウド上に1つだけあり、 複数の開発者がプライベートなローカルリポジトリを分散して持っているというシステムになっているので、 git は「分散型バージョン管理システム」と呼ばれます。
