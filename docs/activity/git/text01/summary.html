アクティビティ: git によるバージョン管理
学習項目: [1] git とは

まとめ

1. バージョン管理システムと git
  • バージョン管理システムを使うとファイルのバージョン管理が楽になる
  • バージョン管理システムを使うとチーム開発が楽になる
  • git がよく使われている
  • git の作者はリーナス・トーバルズ

2. ローカルリポジトリとリモートリポジトリ
  • git を使うと、ユーザーはあるファイルを更新した時に、その時点におけるファイルの内容を「リポジトリ」と呼ばれるデータベースに保存できる
  • ローカルリポジトリ: 開発者の手元のPC内にあり、その開発者がひとりで利用するためのプライベートなリポジトリ
  • リモートリポジトリ: クラウド上にあり、開発チームに所属する複数の開発者が共同利用するリポジトリ

3. git によるチーム開発の流れ

(ステップ1) ある開発者は、自分に割り当てられた作業中の適当なタイミング(※1)でローカルリポジトリに対して更新したファイルをコミット(※2)する。

※1 例えばある程度作業が進んだとか終業時間が来たりとかしたとき。
※2 commit・・・更新したファイルの内容をローカルリポジトリに保存すること。一度コミットすればいつでもコミットした時点の状態にファイルを戻す事ができる。

(ステップ2) その開発者は、割り当たった作業が全て完了するまでステップ1を繰り返す。

(ステップ3) 自分の作業が終わったら、その開発者はリモートリポジトリに対してプル(※3)を実行する。

※3 pull・・・他の開発者が更新してリモートリポジトリにアップしたファイルを自分が作業中のフォルダにダウンロードして取り込むこと。

(ステップ4) 衝突(※4)が起きたら修正する。

※4 conflict・・・他の開発者の更新内容と自分の更新内容が被ってバッティングすること。

(ステップ5) 衝突が無くなるまでステップ3と4を繰り返す。

(ステップ6) リモートリポジトリに対してプッシュ(※5)を実行する。

※5 自分が更新したファイルをリモートリポジトリにアップすること

(ステップ7) ステップ1に戻り新しい作業を開始する。


4. ワーキングツリー/HEAD/インデックス
  • ワーキングツリー ・・・ 実際にユーザーが作業しているフォルダのことでワークツリーとも言う
  • HEAD ・・・ ローカルリポジトリの最新コミットを示す識別子
  • インデックス ・・・ コミットする前に更新したファイルを一時保存しておく場所のことでステージとも言う
