演習

レポート( 兼 詳細仕様書 兼 チェックシート)

※ マスターはスプリント開始時にこのファイルをDLし、スプリント番号を記入したら、スプリントプランニングで内容を記入する。

第1スプリントが始まる前の準備

(1) 各メンバーは演習用のリモートリポジトリを clone する(アドレスは演習時に知らせる)。

(2) マスターはリリースプランニングで作成した仕様書をローカルリポジトリに移動する。

(3) マスターは仕様書を add/commit/push する。

(4) 各メンバーは git pull して仕様書をダウンロードしておく。

(5) マスターはredmine 「ソフト開発演習xxxx」の下に「チーム(番号)」というサブプロジェクトを作る。

(6) マスターは仕様書を見ながらプロダクトバックログを redmine に入力していく。チケットの題名は

S(番号).PB: 内容

とする(Sから:までは半角で入力する。Sはスプリント、PBはプロダクトバックログの略)。

スプリントプランニング

(1) 全員で全体計画を見直す。

(2) もし計画変更があればマスターは全体仕様とプロダクトバックログを修正して git に add/commit/push する。

(3) もしプロダクトバックログに変更があれば redmine のチケットも修正しておく。

(4) 全体仕様書とプロダクトバックログをもとにして、現在のスプリントで行う内容の詳細仕様書とスプリントバックログを作成する。

(5) マスターは詳細仕様書を git に add/commit/push する。

(6) マスターはスプリントバックログを redmine に入力していく。チケットの題名は

S(番号).SB: 内容

とする(Sから:までは半角で入力する。Sはスプリント、SBはスプリントバックログの略)。


デイリースクラムミーティング

(1) 順番に自分の作業の進行状況を報告する。

(2) 各自の現情報告後、もしスコープを逸脱してる人(＝余計な作業をしてる人)がいたらマスターは止めるように指示する。また、遅れている作業がある場合は作業が進んでいる人に手伝うようマスターが指示する。

(2) 当日の各自の作業内容を再確認して作業開始

開発作業

テキストに書いてある項目に注意しながら作業を進める。

原則:

「常にテストする」

「常に動く状態に保つ」

「リモートリポジトリにバグ入りコードをpushしない」


スプリントレビュー、振り返り

テキストに書いてある項目に注意しながら作業を進める。

(1) (レビュー) 実装を予定していたが出来なかった機能、バグ項目などを感情を交えず機械的にリストアップしてドキュメントにまとめる。

(2) (振り返り) 反省点や次のスプリントで追加したい機能、優先して修正するバグなどを話し合ってドキュメントにまとめる。

リリース

(1) 実行ファイル、画像・音楽ファイル、マニュアル等を一つの zip に圧縮する(ソースコードは含めない)。ファイルサイズはアップローダの都合により5M以内に収めること。またファイル名は

T(番号)S(番号).zip

とする(半角で入力。Tはチーム、Sはスプリントの略。例えばチーム 4 の第 2 スプリントのzipなら T4S2.zip )。

(2) zip ファイルを別のフォルダにコピーして解凍し、きちんとプレイ出来ることを確認する。プレイできなかったら(1)に戻る。

(3) zip ファイルをサーバにアップロードする。アップの仕方は演習中に知らせる。

クロスレビューによるユーザー評価

クロスレビュー表

・あるチームが別のチームのゲームをDLして一斉にプレイし、星(1〜5の整数、高い程良い)とコメントをチーム内で話し合って記入する。

・プレイ時間は 5分/ゲーム とする。
