演習 2-1 (チーム、10分): 要求定義の重要性についてチームで調べてみましょう。

「顧客が本当に必要だったもの」

というタイトルの有名なイラストがあります。
このイラストを画像検索をして見つけ、なぜこのような食い違いが起こったのかその理由をみんなで考えて、指定場所にその理由を記述して下さい。


演習 2-2 (チーム、20分): 要求仕様書を作成しましょう。

代表者はプロジェクトフォルダの中に「要求仕様書.docx」を新規作成して下さい。フォーマットは任意です。
開発チーム(自分達)がユーザー(自分達)に対してヒアリングを行い、どういうゲームをユーザーが作って欲しいと思っているのかを理解して箇条書きやイラスト等にまとめて下さい。なお

  • フォーマットは任意です。
  • 大雑把な内容で構いません。
  • 実際に出来るかどうかはこの時点では考えなくて良いです。

ユーザー(自分達)にゲームの名前を決めてもらい、要求仕様書に記述して下さい。


演習 2-3 (チーム、30分): 今回の開発で利用する開発環境を使うとどの様なゲームを作成できるかチームで調べてみましょう。

利用する開発環境を使って作ったゲームを公式サイトなどから見つけます。
見つけたゲームの中から、これから開発するゲームに近いジャンルのゲームを見つけます。
そのゲームを実際にプレイし、この開発環境を使うとどのレペルのゲームを作れるのか確認します。
ソースコードも配布していたら、参考にするためダウンロードしておきます。
指定場所に「ゲームのスクリーンショット」「URL」を記述して下さい。


演習 2-4 (チーム、45分): 全体仕様書を作成しましょう。

代表者は全体仕様書のテンプレートファイル(zentai.docx)をプロジェクトフォルダ内にアップロードして「全体仕様書.docx」に名前を変更して下さい。
全員で話し合い、要求仕様書を元に全体仕様書を作成して下さい。

  • 要求定義で出てきた荒唐無稽な内容が実現可能か吟味し、無理そうなら除外します。
  • ウォーターフォールと違って細かい仕様まで決める必要はありませんが(※)、要求定義の時よりは具体的な内容にして下さい。
  • キャラクターやステージ等のイラストは簡易的なもので構いません。

※ プログラムする際に必要となる細かい仕様はスプリントに入ってから決めます。


演習 2-5 (チーム、45分): 企画書を作りプレゼンしましょう。

代表者は企画書のテンプレートファイル(kikaku.pptx)をプロジェクトフォルダ内にアップロードして「企画書.pptx」に名前を変更して下さい。
全員で話し合い、全体仕様書を元に企画書を作成して下さい。
発表者と記録係を選びます。
発表者はプロジェクタに映した企画書をプレゼン参加者に 1 分以内で説明します。
2 分間ディスカッションします。ディスカッション内容は記録係が記録します。
全てのチームのプレゼンが終わったら、ディスカッションで出た質問や意見を全体仕様書に反映させます。


演習 2-6 (チーム、30分): リモートリポジトリを新規作成してメンバーを登録しましょう。

こちらを参考にして、代表者は GitHub 上に新規でリモートリポジトリを作成します。名前はゲームのタイトルに合わせて下さい。
代表者は端末を開いてリモートリポジトリを clone します。
代表者はこちらの設定ファイル(※)をダウンロードしてローカルリポジトリの中に保存して下さい。

※ gitignore と呼ばれるファイルで、中に名前が書いてあるファイルは git の管理外に置かれます。

代表者はローカルリポジトリの中に移動して以下のコマンドを実行して下さい。 

mv dot.gitignore .gitignore
git add .gitignore
git status
git commit -m "gitignore 追加"
git push

※ git push でIDとパスワードを聞かれたら入力する

再び GitHub に戻り、 こちらを参考にして代表者は他のチームメンバーにインビテーションを送ってコラボレーターとして登録をします。
各メンバーはリモートリポジトリを clone します。
代表者はリモートリポジトリへのリンクをプロジェクトフォルダ内に作成して下さい。リンク名は「リモートリポジトリ.url」とします。


演習 2-7 (チーム、45分): Greenfoot で新規シナリオを作成してリモートリポジトリに登録しましょう。

A〜Dさん役を決めます。
AさんはGreenfootを起動して新規シナリオを作成します。その際にシナリオフォルダとして演習 2-6 で作成したローカルリポジトリを指定します。
Aさんは「World」のサブクラスとして「MyWorld」を作って下さい。キャンパスの大きさ、背景画像等は後で変更出来ますので今の所任意で構いません。
一度 Run して下さい。
Aさんは端末を開き、ローカルリポジトリの中に移動します。
Aさんは「Greenfootを閉じてから」 

git status
git add .
git status
git commit -m "新規シナリオ作成"
git push

の順にコマンドを実行してローカルリポジトリとリモートリポジトリを更新して下さい。
B〜Dさんも端末を開き、ローカルリポジトリの中に移動します。
B〜Dさんは git pull してローカルリポジトリを最新状態に更新して下さい。


演習 2-8 (チーム、45分): リポジトリの同時編集の練習をします。各自で自分のキャラクターを追加してましょう。

全員 greenfoot でシナリオを開いて下さい。
全員 Actor のサブクラスとして適当なキャラを作って下さい。名前、画像は適当で結構です。またソースの中身は空で結構です。
前の作業が終わった人は一度 Run して下さい。
前の作業が終わった人は「Greenfootを閉じてから」 

git status
git add .
git status
git commit -m "(自分の記号(A〜D))がキャラ追加"

の順にコマンドを実行してローカルリポジトリを更新して下さい。
前の作業が終わった人から順次 git pull を実行して下さい。もし画面に

CONFLICT (content): Merge conflict in project.greenfoot

と表示された時は Greenfoot の設定ファイル(project.greenfoot)に衝突が生じましたので Greenfoot を起動して下さい。すると Greenfoot が自動的に設定ファイルを修正します。なおキャラ画像の指定が外れている場合がありますので、その場合は右クリック → Setimage から画像を指定し直します。

project.greenfoot を修正したら「Greenfootを閉じてから」 

git add project.greenfoot
git status
git commit -m "project.greenfoot修正"
git pull

を実行し、また衝突が生じたら再び Greenfoot を起動して設定ファイルを修正して下さい。
衝突が無くなったら git push を実行してリモートリポジトリを修正して下さい。
全員の git push が終わったら、全員が再び git pull してリポジトリを最新状態に更新して下さい。


演習 2-9 (チーム、45分): 次は１つのキャラクターを同時に編集してみましょう。

全員は再びGreenfootを開き、Aさんが作ったキャラに対して同時編集をします。
A さんは↑を押したら上に移動する処理を書いて動作確認して下さい。
B さんは→を押したら右に移動する処理を書いて動作確認して下さい。
C さんは↓を押したら下に移動する処理を書いて動作確認して下さい。
D さんは←を押したら左に移動する処理を書いて動作確認して下さい。
前の作業が終わった人は「Greenfootを閉じてから」 

git status
git add .
git status
git commit -m "(自分の記号(A〜D))がキャラ移動処理を追加"

の順にコマンドを実行してローカルリポジトリを更新して下さい。
前の作業が終わった人から順次 git pull を実行して下さい。もし画面に

CONFLICT (content): Merge conflict in (衝突ファイル名)

と表示されたら衝突が置きましたので修正します。

もし衝突が起きたファイルが設定ファイル(project.greenfoot)の場合は演習 2-8 に従って自動修正します。

設定ファイル以外のファイルで衝突が生じた時は手作業で修正する必要があります。
まず git status を実行し、「both modified」マークが付いているファイルを確認します。
次に Greenfoot を起動し、エラーを修正して動作確認します。
正常に動作したら 

git status
git add .
git status
git commit -m "衝突したファイルを修正"
git pull

を実行し、また衝突が生じたら再び Greenfoot を起動して衝突したファイルを修正して下さい。
衝突が無くなったら git push を実行してリモートリポジトリを修正して下さい。
全員の git push が終わったら、全員が再び git pull してリポジトリを最新状態に更新して下さい。


演習 2-10 (チーム、30分): プロダクトバックログを作成しましょう。

こちらを参考にして、代表者はリモートリポジトリの Projects タブを開き、「プロダクトバックログ」という名前でカンバン(プロジェクト)を新規作成します。
他のメンバーもプロダクトバックログを表示し、 全体仕様書を元にチーム内で話し合いながらバックログアイテムを作成・登録していきます。※ 担当者名は入れなくて良いです。
代表者はプロダクトバックログへのリンクをプロジェクトフォルダ内に作成して下さい。リンク名は「PBL.url」とします。


3. スプリント


演習 2-11 (チーム、30分): スプリントプランニングミーティングを行いましょう。

代表者はプロジェクトフォルダ内に「スプリント(スプリント番号)」というフォルダを作成します。(例) スプリント2なら「スプリント2」
代表者は最新の全体仕様書をスプリントフォルダ内にコピーします。
代表者はコピーした全体仕様書の名前を「全体仕様書(スプリント番号).docx」に変更し。(例) スプリント2なら「全体仕様書2.docx」
全員がコピーした全体仕様書を開き、話し合って全体仕様書を見直します。スプリント2以降なら前のスプリントで実施したレトロスペクティブやユーザーからのフィードバックも参考にして下さい。
「プロダクトバックログ」のToDo列に含まれるバックログアイテムの修正も合わせて行います。
このスプリントで実施するバックログアイテムを選択し、Doing列に移動します。


演習 2-12(チーム、30分): 詳細仕様書を作成しましょう。

代表者はスプリントフォルダ内に「詳細仕様書(スプリント番号).docx」を作成します。フォーマットは任意です。(例) スプリント2なら「詳細仕様書2.docx」
みんなで話し合って今回のスプリントで実装予定の内容を文章や箇条書き、クラス図、表、グラフ、イラスト等で詳しくまとめて下さい。

  • フォーマットは任意です。
  • 画像のサイズとかキャラクターの画像や移動スピードなど、かなり細かい数値まで具体的に決めて下さい
  • 仕様は逐次見直し可能なので、深く考えず素早く仕様を決めて下さい
  • クラス図やシーンの遷移図、画面等のイラストがあると今後の開発が楽になります
  • イラストは再利用可能な様に手描きではなくてドローツールやイラストレイター等を使って作成して下さい
  • ネットにある素材を使用する場合は必ずライセンスを確認して下さい。もしフリー素材を使用する場合は必ずアドレスも貼り付けて下さい

演習 2-13 (チーム、10分): スプリントバックログを作成しましょう。

こちらを参考にして、代表者はリモートリポジトリの Projects タブを開き、「スプリントバックログ(スプリント番号)」カンバンを新規作成します。(例) スプリント2なら「スプリントバックログ2」
他のメンバーも作成したカンバンを表示し、 詳細仕様書を元にチーム内で話し合いながらスプリントバックログアイテムを作成・登録していきます。その際に作業担当者も決めて名前(個人特定されないように要注意)も記入して下さい。
代表者はプロダクトバックログへのリンクをプロジェクトフォルダ内に作成して下さい。リンク名は「SBL(スプリント番号).url」とします。(例) スプリント2なら「SBL2.url」


演習 2-14 (チーム、30分): デイリースクラムミーティングを行いましょう。

相互に現状報告をしあってください。
相互にスコープ確認しあってください。
今後のスケジュールを確認しあってください。
もし詳細仕様の変更が必要なら、全員で話し合って詳細仕様書とスプリントバックログを更新して下さい。
もし全体仕様の変更が必要なら、全員で話し合って全体仕様書とプロダクトバックログを更新して下さい。
スプリントバックログを見て相互に本日の作業内容を確認しあってください。


演習 2-15 (チーム、30分): 開発作業を行いましょう。

自分の担当している作業を行います。
適当なタイミングで 

git status
git add .
git status
git commit -m "(適当なコメント)"

を実行してローカルリポジトリを更新します。
作業が終わったら git pull を実行し(もし衝突したら修正して下さい)、git push して下さい。

※ マナーとして git push するときは他のメンバーに「これから push する」と一声かけましょう。

git push したら対応するスプリントバックログも忘れず Done 状態にして下さい。
プロダクトバックログも、対応するタスクが全て終わったら適宜 Done に更新して下さい。


演習 2-16 (チーム、30分): スプリントレビューを行いましょう。

以下のチェック項目が全て満たされている事を確認して下さい。

スプリントバックログが全て Done 状態になっている(※)
今回のスプリントで実施するプロダクトバックログが全て Done 状態になっている(※)
リモートリポジトリにあるプロジェクトにはビルドエラーが起きない
リモートリポジトリにあるプロジェクトにはバグが無い
ライセンス違反をしていない
人格権違反や個人情報の漏洩をしていない

※ 時間内に Done 状態まで辿りつけなかったタスクは次のスプリントに回して下さい。


演習 2-17 (チーム、10分): レトロスペクティブを行いましょう。

代表者はスプリントフォルダ内に「レトロスペクティブ(スプリント番号).docx」を作成します。フォーマットは任意です。(例) スプリント2なら「レトロスペクティブ2.docx」
全員で反省点や全体仕様書の見直し(※)などを話し合い、話し合った内容を記述して下さい。

※ ちゃんとした見直し作業はユーザーレビューも踏まえて次のスプリントのプランニングミーティングで行いますので、ここでは見直し項目を挙げる程度の簡単な作業で結構です。


演習 2-18 (チーム、5分xチーム数): リリース作業を行いましょう。

代表者は最新リポジトリをzipで固め、スプリントフォルダ内にアップして下さい。ファイル名は「(班番号)-(スプリント番号).zip」として下さい。(例) 3班、スプリント2なら 「3-2.zip」


演習 2-19 (チーム、5分xチーム数): ユーザーからフィードバックをもらいましょう。

代表者はアンケートフォームを作成します。

タイトルは「レビュー(班番号)-(スプリント番号)」とします。(例) 3班、スプリント2なら 「レビュー3-2」

アンケート項目は以下の通りです。項目の説明文は任意とします。

  1. 星(5段階) ※ 必須
  2. 自由記述コメント(1行) ※ 任意

スプリントフォルダ内にアンケートへのリンクを作成して下さい。リンク名は「レビュー(班番号)-(スプリント番号).url」とします。(例) 3班、スプリント2なら 「レビュー3-2.url」
アンケートの設定を開いて匿名設定にしてから回答可能状態にします。
クロスレビューを行います。各自は指定されたチームのゲームをダウンロードして3分ほどプレイし、アンケート項目に答えてください。
レビューが終わったら回答の受け入れを終了し、結果(xlsx形式)をダウンロードしてスプリントフォルダにアップして下さい。

準備中
