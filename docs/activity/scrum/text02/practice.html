2. リリースプランニング


演習 2-1 (チーム、10分): (シチュエーション: スクラムマスター主催の勉強会) 要求定義の重要性についてチームで調べてみましょう。

要求定義の重要性を表している

顧客が本当に必要だったもの

というタイトルの有名なイラストがあります。
このイラストを画像検索をして見つけ、なぜこのような食い違いが起こったのかその理由をみんなで考えて、指定場所にその理由を記述して下さい。


演習 2-2 (チーム、20分): (シチュエーション: 顧客との打合せ) 要求定義をおこないます。

チームに1枚ホワイトボードを配布します。自由に使って構いません。
開発チーム(自分達)がユーザー(自分達)に対してヒアリングを行い、どういうゲームをユーザーが作って欲しいと思っているのかを理解して箇条書きやイラスト等にまとめて下さい。なお

  • 大雑把な内容で構いません。
  • 実際に出来るかどうかは考えなくて良いです。

ユーザー(自分達)にゲームの名前を決めてもらいます。
指定場所に「ゲームの名前」「ユーザーが要求した内容」を記述して下さい。イラスト等を描いた場合は写真も撮って貼り付けて下さい。


演習 2-3 (チーム、30分): (シチュエーション: スクラムマスター主催の勉強会) 今回の開発で利用する開発環境を使うとどの様なゲームを作成できるかチームで調べてみましょう。

利用する開発環境を使って作ったゲーム一覧ページを公式サイトなどから見つけます。
それらゲームの中から、これから開発するゲームに近いジャンルのゲームを手分けして見つけます。
そのゲームを実際にプレイし、この開発環境を使うとどのレペルのゲームを作れるのか確認します。
ソースコードも配布していたら、参考にするためダウンロードしておきます。
指定場所に「ゲームのスクリーンショット」「URL」を記述して下さい。


演習 2-4 (チーム、45分): (シチュエーション: 開発ルームでの作業) 要求定義を元に全体仕様書を作成します。

要求定義を元に、作成するゲームの全体仕様を文章や箇条書き、表、グラフ、イラスト等にまとめて下さい。

  • 要求定義で出てきた荒唐無稽な内容が実現可能か吟味し、無理そうなら除外します。
  • WFと違って細かい仕様まで決める必要はありませんが(※)、要求定義の時よりは具体的な内容にして下さい。
  • 簡潔で良いのでシーンの遷移図や画面等のイラストがあると今後の開発が楽になります。
  • イラストは再利用可能な様に手描きではなくてドローツールやイラストレイター等を使って作成して下さい。

※ プログラムする際に必要となる細かい仕様はスプリントに入ってから決めます。

指定場所に「全体仕様」を記述して下さい。表やグラフ、イラスト等を作成した場合は画像ファイルに変換して貼り付けて下さい。


演習 2-5 (チーム、45分): (シチュエーション: 重役の前で社内プレゼン) 全体仕様書を元にプレゼンを行い、ディスカッションで出た質問や意見を全体仕様書に反映させます。

発表者と記録係を選びます。
発表者はプロジェクタに映した全体仕様書を他の参加者に 1 分以内で説明します。
2 分間ディスカッションします。ディスカッション内容は記録係が記録します。
全てのチームのプレゼンが終わったら全体仕様書の見直しを行います。


演習 2-6 (チーム、30分): (シチュエーション: 開発ルームでの作業) リモートリポジトリを新規作成してメンバーを登録します。

こちらを参考にして、代表者は GitHub 上に新規でリモートリポジトリを作成します。名前はゲームのタイトルに合わせて下さい。
代表者は端末を開いてリモートリポジトリを clone します。
代表者はこちらの設定ファイル(※)をダウンロードしてローカルリポジトリの中に保存して下さい。

※ gitignore と呼ばれるファイルで、中に名前が書いてあるファイルは git の管理外に置かれます。

代表者はローカルリポジトリの中に移動して以下のコマンドをコピーして実行して下さい。 

mv dot.gitignore .gitignore
git add .gitignore
git commit -m "gitignore 追加"
git push

※ git push でIDとパスワードを聞かれたら入力する

再び GitHub に戻り、 こちらを参考にして代表者は他のチームメンバーにインビテーションを送ってコラボレーターとして登録をします。
各メンバーはリモートリポジトリを clone します。
指定場所にリモートリポジトリのトップアドレスを記述して下さい。


演習 2-7 (チーム、30分): (シチュエーション: 開発ルームでの作業) プロダクトバックログを作成します。

こちらを参考にして、代表者はリモートリポジトリの Projects タブを開き、「プロダクトバックログ」という名前でカンバン(プロジェクト)を新規作成します。
他のメンバーも作成したカンバンを表示し、 全体仕様書を元にチーム内で話し合いながらバックログアイテムを作成・登録していきます。※ 担当者名は入れなくて良いです。
指定場所にカンバンのアドレスを記述して下さい。


3. スプリント


演習 2-8 (チーム、30分): (シチュエーション: 開発ルームでの作業) スプリントプランニングミーティングを行います。

(必要なら)全体仕様書を見直し、「プロダクトバックログ」カンバンのToDo列に含まれるアイテムの修正を行います。
今回のスプリントで実施するプロダクトバックログアイテムを選択し、カンバンのDoing列に移動します。
詳細仕様書を作成します。
指定されたファイルに今回のスプリントで実装予定の内容を文章や箇条書き、クラス図、表、グラフ、イラスト等に詳しくまとめて下さい。

  • 画像のサイズとかキャラクターの移動スピードなど、かなり細かい数値まで決めて下さい。
  • この後のデイリースクラムミーティングで仕様は見直し可能なので、深く考えず素早く仕様を決めて下さい。
  • クラス図やシーンの遷移図、画面等のイラストがあると今後の開発が楽になります。
  • イラストは再利用可能な様に手描きではなくてドローツールやイラストレイター等を使って作成して下さい。

代表者はリモートリポジトリの Projects タブを開き、「スプリントバックログ○」(※)という名前でカンバン(プロジェクト)を新規作成します。

※ ○の中にはスプリント番号を入れて下さい。 (例) スプリント2

他のメンバーも作成したカンバンを表示し、 詳細仕様書を元にチーム内で話し合いながらバックログアイテムを作成・登録していきます。
担当者も決めて名前(個人特定されないように注意)を記入して下さい。


演習 2-9 (チーム、45分): (シチュエーション: スクラムマスター主催の勉強会) Greenfoot による共同開発の流れ(新規シナリオ作成→同時編集→衝突処理)を練習してみましょう。

A〜Dさん役を決めます。
端末を開いて、全員リモートリポジトリの中に移動します。
AさんはGreenfootを起動し、「SCR2-9」という名前の新規シナリオをリモートリポジトリ内に作ります。
Aさんは「World」のサブクラスとして「MyWorld」を作って下さい。キャンパスの大きさ、背景画像は任意です。
一度 Run して下さい。
AさんはGreenfootを閉じ、Greenfootが作成したファイルを全て add → commit (コメントは任意) → push してリモートリポジトリにアップして下さい。
B〜Dさんは pull してリポジトリを最新状態に更新し、SCR2-9 フォルダに移動して下さい。
全員は A さんが作ったシナリオを開いて下さい。
全員は「Actor」のサブクラスとして適当なキャラを作って下さい。
名前、画像は適当で結構です。
またソースの中身は空で結構です。
全員一度 Run して下さい。
作業が終わった人から順次Greenfootを閉じ、git status で状態を確認してから更新されたファイルを全て add → commit (コメントは任意)して下さい。
終わった人から順次 pull して下さい。
(以下、衝突処理) 「CONFLICT (content): Merge conflict in project.greenfoot」と表示された時は Greenfoot の設定ファイル(project.greenfoot)に衝突が生じましたので Greenfoot に修正してもらいます。
Greenfoot を起動してそのまま閉じて下さい(※)。

※ キャラ画像の指定が外れている場合があります。その場合は右クリック → Setimage から画像を指定し直して下さい。

git add project.greenfoot してから commit (コメントは任意) して下さい。
再び pull してまた衝突が生じたら上に戻って下さい。 (衝突処理ここまで)
(衝突が無くなったら) push して下さい。
全員の push が終わったら、全員再び pull してリポジトリを最新状態に更新して下さい。
全員は再びGreenfootを開き、Aさんが作ったキャラに対して同時編集をします。
A さんは↑を押したら上に移動する処理を書いて動作確認して下さい。
B さんは→を押したら右に移動する処理を書いて動作確認して下さい。
C さんは↓を押したら下に移動する処理を書いて動作確認して下さい。
D さんは←を押したら左に移動する処理を書いて動作確認して下さい。
先ほどと同じように終わった人から順次Greenfootを閉じ、更新したファイルをadd → commit → pull して下さい。
(以下、衝突処理) 「CONFLICT (content): Merge conflict in 〜」と表示された時はそのファイル内で衝突が生じましたが、設定ファイル以外は手作業で修正する必要があります。
git status を実行し、「both modified」マークが付いているファイルを確認します。
Greenfoot を起動し、エラーを修正して動作確認します。
修正したファイルを全て add してから commit (コメントは任意) して下さい。
再び pull してまた衝突が生じたら上に戻って下さい。 (衝突処理ここまで)
(衝突が無くなったら) push して下さい。
全員の push が終わったら、全員再び pull してリポジトリを最新状態に更新して下さい。


演習 2-10 (チーム、30分): (シチュエーション: 開発ルームでの作業) デイリースクラムミーティングと開発作業を行います。

テキストを参考にデイリースクラムミーティングを実施して下さい。
テキストを参考に開発作業に入って下さい。
作業が一区切りしたら git add → commit → pull → push して下さい(※)。またスプリントバックログも Done に更新して次の作業に移って下さい。

※ push するときは他のメンバーに「これからpushする」と一声かけましょう。

まだ作業途中でも、演習が終わる前に一度 git add して状態を保存しておいてください。


演習 2-11 (チーム、30分): (シチュエーション: 開発ルームでの作業) スプリントレビューを行います。

以下のチェック項目が全て満たされている事を確認して下さい。

スプリントバックログが全て Done 状態になっている(※)
リモートリポジトリにあるプロジェクトにはビルドエラーが起きない
ライセンス違反をしていない
人格権違反や個人情報の漏洩をしていない
リモートリポジトリにあるプロジェクトにはバグが無い

※ 時間内に Done 状態まで辿りつけなかったタスクは次のスプリントに回して下さい。

全て Done 状態になっているスプリントバックログのスクリーンショットを指定された場所に貼り付けて下さい。


演習 2-12 (チーム、10分): (シチュエーション: 開発ルームでの作業) レトロスペクティブを行います。

「反省点」「全体仕様書の見直し(※)」などを話し合い、話し合った内容を指定された場所に記述して下さい。

※ ちゃんとした見直し作業はユーザーレビューも踏まえて次のスプリントのプランニングミーティングで行いますので、ここでは見直し項目を挙げる程度の簡単な作業で結構です。


演習 2-13 (チーム、10分): (シチュエーション: 市場) リリースとユーザーレビューを行います。

代表者はプロジェクトフォルダ全体をzipで固め指定した場所にアップして下さい。ファイル名は「○班.zip」として下さい。
それと同時に別の代表者はアンケート用のフォームを作成し、指定した場所にリンクを貼り付けて下さい。
クロスレビューを行います。各自は指定されたチームのゲームをダウンロードしてプレイし、指定された場所でレビューを行って下さい。
