WBS の一番楽な作り方は、所属する組織が過去のプロジェクトで作った WBS をテンプレートにして再利用することです。
あるいは有料・無料を問わず WBS 作成ツールがいっぱいありますのでそれらを利用することです。

たたテンプレートが無かったり、ツールを使いたくない場合は仕方がないので WBS をゼロから作る必要があります。

まずは以下の手順でプロジェクトに含まれるタスクを全て洗い出します。

  1. プロジェクトの最終目的を一番左(ルート)に書く
  2. ルートのタスクをいくつかの子タスクに分解してルートの右側に列挙し、ルートから線を引く
  3. 2で挙げたタスクを更にいくつかの子タスクに分解して親タスクの右側に列挙し、親から線を引く
  4. 3で挙げたタスクを更にいくつかの子タスクに分解して親タスクの右側に列挙し、親から線を引く
  5. 以上を繰り返す

なお各階層の子タスクの数は 3〜5 個位に留めておくのが良いらしいです(最適な数には諸説あり)。

またタスクは「全てのプロジェクトメンバーが具体的に何をすれば良いのか理解できる作業工程」つまり前のページで出てきた「アクティビティ(ワークパッケージ)」が出てくるまで分解し続けます。
従ってあるタスクで一人でも何をすれば良いのか分からないメンバーがいたら、そのタスクを小タスクに更に分解しなければいけません(※)。

※ とは言え、あまり細かい所までタスクを分解してしまうと担当者に裁量権が無い硬直したプロジェクトになりますので注意が必要です。

アクティビティを洗い出したら、次はアクティビティ間の依存関係を洗い出します。

例えばアクティビティ A を開始するためには アクティビティB が終わってないといけない、アクティビティ B を開始するためには アクティビティ C が終わってないといけないという状況は矢印を使って

C → B → A

のような依存関係で表されます(一種のPERT図になっていることに注目して下さい)。

次に、洗いだしたアクティビティとそれらの依存関係をもとにしてガントチャートを作ります。

ガントチャートは棒グラフの一種で、典型的には最初の列にアクティビティ一覧、次の列からは担当者名や進捗率や予算額などを記入し、最後に各アクティビィティが開始する時期から終了する時期まで横棒を引きます。

なおアクティビティ間の依存関係を表すために横棒と横棒の間に矢印を引くことも多いです。
もちろんアクティビティ間に依存関係がある場合は、前のアクティビティが終了するまでは後のアクティビィティを開始できません。

ただ、依存関係みたいな面倒な縛りがあるからと言って適当にスケジューリングすると「作業終了待ちで暇なメンバー」が生じますので、可能な限りアクティビティを並行で動かして暇なメンバーが生じないよう効率的にスケジューリングする必要があります。

ところで最も効率よくスケジューリングすると「依存関係のせいで並列作業出来ないアクティビティの最短の繋がり(パス)」がガントチャート上に現れます。
そのパスのことをクリティカルパスと呼び、クリティカルパス上のアクティビティの合計時間がプロジェクトを完遂するための理論的な最小時間になります。

いずれにしろプロジェクトメンバーが一目見ただけですぐ理解出来るくらい分かりやすい WBS を描くことが重要です。
適宜コメントを入れたり、カレンダーと連動させたり、横棒を色分けしたりするなど、創意工夫している WBS も良く見かけます。
