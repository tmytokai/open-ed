「スクラム開発」は反復型開発手法の一種で、オブジェクト指向型プログラミングを用いたソフトウェア開発ととても相性が良い手法なので広く使われています。

「スクラム」はソフトウェア開発工程を「スプリント」と呼ばれる複数の開発期間(一般的に1〜4週間程度で期間中に開発からリリースまで実施する)に分け、そのスプリントを繰り返し実行することでソフトウェアを完成させるという反復型開発手法です。
本格的なスクラム演習はこちらで行うことにし、今回は簡易化したスクラム(※)を使ってミニゲームを開発することでオブジェクト指向開発の流れを実際に体験してみようと思います。

さて、このページではまず「リリースプランニング(全体計画)」を行います。
リリースプランニングはスクラムの最初に実施する作業工程で、要求定義、全体仕様書の作成、プロダクトバックログや WBS の作成を行います。

1. 要求定義

「要求定義」とは、クライアントがソフトウェアを使って何をしたいのかを明確にするためにソフトウェア開発の初期段階でおこなう作業のことです。
具体的には「要求仕様書」というドキュメントを作成します。

要求仕様書の書き方には特に決まり事はないのですが、最低限でも以下の項目は含めた方が良いと思います。

  • 名称
  • 用途・目的
  • 対象ユーザ
  • システム構成
  • 概要

また図などが含まれていると分かりやすくなるのでより良いと思います。

さて今回の演習では以下の様な要求仕様書を作成しました。

要求仕様書

名称TARO と HANAKO
用途・目的暇つぶし用ミニゲーム
対象ユーザ6 歳以上の男女
システム構成 OS: Windows 又は Linux
PC: 低スペック
開発環境: Greenfoot 又は Unity2D
概要 ・PC 上で動く 2 次元アクションゲーム
・キーボードのみで操作
・画面上に TARO と HANAKO というプレイヤーキャラクターと複数の果物を表示する
・ユーザーはキーボード操作により TARO と HANAKO を動かせる
・TARO が果物に触れたらゲームオーバー
・HANAKO が制限時間内に画面上の全ての果物を食べたらゲームクリア
・動きまわる果物もある


2. 全体仕様書の作成

次に「全体仕様書」を作成します。

全体仕様書は要求定義書を元に作成した全体の大まかな仕様書のことで、曖昧だった部分をある程度詳細までつめてドキュメント化していきます。

全体仕様書の書き方も特に決まり事はないのですが、まず以下の様にゲーム仕様を策定してみました。

ゲーム仕様

・背景(MyWorld と名付ける)は一枚絵とし、スクロールしない
・ゲーム開始時の TARO の初期位置は画面左上
・ゲーム開始時の HANAKO の初期位置は画面右下
・TARO はカーソルキーで上下左右に動かせる
・HANAKO もカーソルキーで上下左右に動かせるが動きは逆になる(例えば上を押すと下に移動する)
・TARO と HANAKO 間に当たり判定は無い(スレ違い可能)
・TARO と HANAKO は画面外に移動できない
・果物には APPLE と ORANGE がある
・ゲーム開始時に APPLE と ORANGE をランダムな位置に複数個表示する
・APPLE は動きまわる
・ORANGE は静止している
・TARO が果物に触れたらゲームオーバーとなりメッセージ表示
・HANAKO が果物に触れたら食べたことにして果物を消す
・HANAKO が画面上全ての果物を食べたらゲームクリアとなりメッセージ表示
・時間制限を設け、時間内に全ての果物を消せなかったらタイムアップとなりメッセージ表示
・残り時間は画面左下に表示

なおウォーターフォールの場合は後戻りできないので、この仕様策定の段階で画面サイズとかメッセージ内容とかの細かい項目まで全て完全に決める必要があります。
一方、スクラムの場合は細かい項目に関してはスプリントの中で柔軟に決める事が多いですし、そもそも仕様変更が頻繁に起きますので、仕様策定段階ではそこまできっちり決める必要は無いと思います。

ところで上のような文章だけだと分かりにくいので図も全体仕様書に追加すると良いと思います。
例えば以下の様な画面仕様図を描いてみました。

画面仕様図

またこの段階でクラス設計も大まかにしておくと今後の見通しが良くなると思います。
詳細なクラス設計はスクラムの最中に行うことにして、今回は次の様なクラス設計をしてみました。

クラス設計

クラス名内容
MyWorld ・背景クラス
・キャラクターの初期配置やタイムアップ処理などのマネージャー処理も行う
HUMAN ・TARO と HANAKO のスーパークラス
・キーボード操作(上下左右移動)を行う
・果物との当たり判定を行い、ゲームクリア処理を行う
TARO ・プレイヤーキャラクラス その 1
・当たり判定をオーバーライドし、果物に触れたらゲームオーバー処理を行う
HANAKO ・プレイヤーキャラクラス その 2
・キーボード操作をオーバーライドし、動きを反転させる
FRUIT ・APPLE と ORANGE のスーパークラス
・特に何もしない
APPLE ・果物クラス その 1
・ランダムに動く
ORANGE ・果物クラス その 2

3. プロダクトバックログと WBS の作成

全体仕様書を元に、作業内容をさらに細かく分割して「タスク」を作成します。
さらに各タスクの優先度、担当者などを決めてリスト化します。
この様にして作成したタスクリスト、またはタスクそのものの事を「プロダクトバックログ」と呼びます。

プロダクトバックログを作成したら各タスクの優先度に従って開発スケジュールを決めます。
その際に「WBS (Work Breakdown Structure: 作業分解構成図)」を利用すると便利です(※)。

※ ちなみに WBS を使ってきっちりスケジューリングするのはウォーターフォール的思考なので止めたほうが良いという意見もあります。

プロダクトバックログや WBS の書き方にも特に決まり事はないのですが、今回はクラス構造とプロダクトバックログとガントチャートを一体化した WBS を以下の様に作成してみました。
左の方から右の方に向かって行くに連れてツリー構造状に項目が細かくなって行き、どの項目がどのスプリントで処理されるのかが一目瞭然で分かるようになっています。

なお、一般的なスプリントの期間は上で書いたとおり1〜4週間なのですが、今回は簡易版スクラムなので 1 スプリント ＝ 20 分とします。
