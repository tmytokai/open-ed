コミットの巻き戻し


あるコミットで致命的なバグが混入した時に、そのコミットの前まで状態を巻き戻したい場合があります。
ここではその方法について学びます。

その前にまず、現在作業中に致命的なミスをやらかしてしまったので、HEADの状態までワーキングツリーとインデックスを強制リセットする方法について学びます。
出来れば使いたくないコマンドですが、実際の開発中は意外に良くお世話になります。 

HEADの状態までワーキングツリーとインデックスを強制リセット( やや危険 )


 git reset --hard HEAD


※ HEAD 後に更新した内容が完全に全て消えるので実行する時は要注意
　ファイル内容の復元のテキストで学んだ
  git checkout HEAD -- .
  を使う方法と異なって add した新規ファイルも全て消えます

次にこのページの本題であるコミットの巻き戻しについて学びます。
まずは安全(だけど面倒)な方法からです。 

N 回コミットを巻き戻す(安全な方法)


 (1) 以下のように revert コマンドを N 回繰り返す


git revert -n HEAD
git revert -n HEAD~1
git revert -n HEAD~2
：
：
git revert -n HEAD~(N-1)


※ 1 回だけ巻き戻したい時は git revert -n HEAD だけで結構です。
※ -n オプションを付けないと(有難迷惑なことに)コミットが実行されます。


 (2) diff コマンドで差分が無いか調べる


git diff HEAD~N


 (3) コミットする


git commit -m "N回巻き戻し"

例えば状態19から N = 3 回巻き戻した状態が以下の状態20となります。
状態20のワーキングツリー、インデックス、HEADが状態19のHEAD~3( = 状態20のHEAD~4)と同じ内容になっていることに注目して下さい。

最後に、危険(だけど簡単)な方法について学びます。 

N 回コミットを巻き戻す(危険な方法)


 git reset --hard HEAD~N


※ 更新した内容が完全に全て消えるので実行する時は要注意

教えておいて何ですが普通は使わない方が無難です。
特にこの後学ぶリモートリポジトリを使って多人数でチーム開発する時は、他のメンバーに大迷惑がかかるのでこの方法は厳禁です。
ちなみに HEAD~N を HEAD にした場合がこのページの最初に学んだ「HEADの状態までワーキングツリーとインデックスを強制リセット(やや危険)」です。

では試しに状態20から N = 3 回この危険な方法で巻き戻してみましょう。
すると状態が以下の状態21となります。
状態が状態 8 まで戻り、その後の更新は無かったことになります。
